# Release 05：Handoff Kit

## 你現在在哪個階段

你已經完成：

- Phase 0 prototype。
- 使用者訪談與需求取捨。
- 流程設計。
- v1 prototype。
- 變更事件分析與必要的穩定化。

接下來進入**交接與維護**。真實軟體不會只由原作者永遠維護；下一位協作者能不能理解你的專案，是工程品質的一部分。

## 這一階段要完成什麼

請完成四件事：

1. 原作者填寫 `docs/handoff.md`。
2. 小隊內交換 repo。
3. 接手者用 Codex 協助理解陌生 repo。
4. 接手者填寫 `docs/handoff-review.md`。

最後每位學員再填寫 `docs/showcase-submission.md`，供小隊 showcase 使用。

## 你會使用哪些檔案

原作者請填：

- `docs/handoff.md`
- `docs/showcase-submission.md`

接手者請填：

- `docs/handoff-review.md`

可使用：

- `docs/prompts/maintainer-onboarding.md`

## 原作者要整理什麼

請讓接手者能快速知道：

- 如何啟動專案。
- demo 入口在哪裡。
- 你優先服務哪一種使用者。
- 你的主要流程是什麼。
- 你做了哪些需求取捨。
- change request 後目前支援、部分支援或不支援什麼。
- 下一位接手者應該先看哪些檔案。
- 哪些地方不能讓 AI 自動判斷。

## 接手者要檢查什麼

請檢查：

- 能不能啟動。
- 能不能理解主流程。
- 能不能找到重要文件與畫面入口。
- 能不能看懂原作者為什麼這樣設計。
- 如果要繼續維護，第一個會卡住的地方是什麼。
- Codex 哪裡幫你理解，哪裡可能誤解。

## Codex 可以做什麼

Codex 可以幫接手者：

- 閱讀 README、handoff、decisions、flow、change-notes。
- 找到 App 入口與主要 component。
- 用白話整理專案目前狀態。
- 找出可能的下一步。
- 找出文件不足或容易誤解的地方。

## Codex 不應該做什麼

Codex 不應該：

- 在接手檢查時直接重寫原作者專案。
- 替接手者決定原作者的設計是對或錯。
- 把未確認資訊當成已確認。
- 自動做救災、派工或行動優先順序判斷。

## 完成條件

完成這一階段時，每位學員應該有：

- 自己的 `docs/handoff.md`。
- 別人寫給自己的 `docs/handoff-review.md` 或等價回饋。
- 自己填寫的 `docs/showcase-submission.md`。

每個小隊應該能整理出：

- 最常見的交接卡點。
- 最有幫助的文件。
- 最容易讓 Codex 誤解的地方。
- 1–2 個代表 demo，供小隊 showcase 使用。
